# EducaOne v2.13.31 — Auditoría completa + Backups + Guía de despliegue

## 1. AUDITORÍA "BIEN DE BIEN" (dos capas)

### Capa 1: Cobertura total (middleware automático)
Un middleware registra AUTOMÁTICAMENTE toda escritura exitosa
(POST/PUT/DELETE/PATCH) sin tener que tocar cada endpoint. Guarda:
quién (usuario_id del token), qué (método + ruta), desde dónde (IP),
cuándo (fecha), y el status. Usa sesión de BD propia para no interferir.
La auditoría NUNCA rompe la respuesta (todo en try/except).

Resultado: NINGUNA escritura queda sin rastro. Antes solo 49 acciones
se auditaban; ahora se auditan TODAS las 117 operaciones de escritura.

### Capa 2: Detalle fino (antes/después) en críticos
Los endpoints más sensibles guardan el detalle completo:
- **Calificaciones secundaria**: guarda nota anterior y nueva (p1-p4,
  rp1-rp4). Podés ver exactamente qué nota cambió y de cuánto a cuánto.
- **Estudiantes**: ya guardaba detalle (crear/editar)
- **Usuarios**: ya guardaba detalle (crear)

Esto es clave con datos de menores: si alguien cambia una nota, queda
registrado quién, cuándo, desde qué IP, y el valor antes/después.

## 2. BACKUP DE DATOS (script adicional)

`backend/backup_datos.py` genera una copia COMPLETA en JSON de todos
los datos, para guardar fuera de Render.

  python backup_datos.py
  → genera backup_educaone_YYYYMMDD_HHMMSS.json

IMPORTANTE: el backup PRINCIPAL lo hace Render automáticamente
(PostgreSQL con backups diarios en planes de pago). Este script es tu
red de seguridad ADICIONAL.

Para backup diario automático: configurá un Cron Job en Render que
ejecute este script.

## 3. DÓNDE DESPLEGAR (recomendación senior)

Arquitectura recomendada:
  Frontend (React)  → VERCEL (gratis, rápido, deploy desde Git)
  Backend (FastAPI) → RENDER (workers, escala bien, backups Postgres)
  Base de datos     → RENDER PostgreSQL (backups diarios automáticos)
  Caché             → RENDER Redis (cuando pases ~20 colegios)

NO uses Vercel para el backend: es para frontend/serverless, no para
FastAPI con conexiones persistentes y workers.

### Backups en Render (lo que activás)
- Plan de PostgreSQL de PAGO (no free) → backups diarios automáticos
- Planes altos: point-in-time recovery
- Adicional: corré backup_datos.py periódico y guardá el JSON aparte

### Por qué Render para el backend
- Backups de Postgres automáticos (clave para datos de menores)
- Workers configurables (WEB_CONCURRENCY)
- Redis disponible como add-on
- Escala con el plan sin cambiar código
